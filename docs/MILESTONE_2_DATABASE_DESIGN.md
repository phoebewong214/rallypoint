# RallyPoint — Milestone 2 Database Design & Implementation

**Deliverable 2 of 3 — Database Design and Implementation**
Author: Phoebe Wang · ACIS 498 Capstone, Northwestern University
Repository: https://github.com/phoebewong214/rallypoint

RallyPoint's data layer is a **SQLAlchemy ORM** model over **15 relational
tables**. It runs on **SQLite in development** and **PostgreSQL in production**
— the same ORM code drives both, switched by a single `DATABASE_URL` environment
variable (no code change). This document is the full schema: every table, its
columns, keys, constraints, indexes, and the relationships between them.

The authoritative `CREATE TABLE` DDL (as generated by the models) is checked in
alongside this file as **`schema_ddl.sql`**.

**Reading the diagram:** `1 ─── *` = one-to-many. `uq(...)` = a unique
constraint (composite where multiple columns are listed). Every `FK` is a
foreign key into the referenced table's primary key.

## 2. Design principles

1. **Per-sport profiles, not per-user ratings.** A player can be a 4.0 tennis
   player *and* a 3.0 pickleball player, so skill/availability/home-court live in
   `sport_profiles` (one row per sport), not on `users`. Enforced by
   `UNIQUE(user_id, sport)`.
2. **Composite unique constraints make writes idempotent.** Favoriting a court,
   saving a player, or checking in twice is a no-op — the `(user, target)`
   unique constraint guarantees at most one row, so the API never accumulates
   duplicates.
3. **Two directed foreign keys for two-party relationships.** A `sessions` row
   points at both `host_id` and `guest_id`; a `game_invites` row at both
   `inviter_id` and `invitee_id`; a `user_reports` row at `reporter_id` and
   `reported_id`. SQLAlchemy disambiguates with explicit `foreign_keys=[...]`.
4. **Invites materialize into sessions.** The two-phase negotiation lives in
   `game_invites` + `time_proposals`; once a time is locked, a real `sessions`
   row is created and referenced by `game_invites.session_id`, so the calendar
   surface only ever reads one table.
5. **Soft-delete over hard-delete for auditable entities.** Suspending a user
   (`users.is_active = false`), closing a court (`courts.is_active = false`), and
   cancelling a session (`status = 'cancelled'`) all preserve the row and its
   history rather than destroying it.
6. **Portability.** Types are chosen to be identical across SQLite and Postgres
   (`INTEGER`, `VARCHAR(n)`, `TEXT`, `FLOAT`, `BOOLEAN`, `DATETIME`), so the one
   `DATABASE_URL` switch is genuinely code-free.

## 4. Indexes & constraints summary

**Unique constraints (business-rule integrity):**

| Table | Unique key | Guarantees |
|---|---|---|
| `users` | `email`, `handle` | One account per email; unique @handle |
| `sport_profiles` | `(user_id, sport)` | One profile per sport per user |
| `availability_slots` | `(user_id, day_of_week, time_band)` | One grid cell per slot |
| `courts` | `slug` | Unique URL id |
| `court_favorites` | `(user_id, court_id)` | No duplicate favorites |
| `court_checkins` | `(court_id, user_id)` | One active check-in per court |
| `appointment_participants` | `(appointment_id, user_id)` | Can't join a game twice |
| `saved_players` | `(user_id, player_id)` | No duplicate bookmarks |
| `ai_match_logs` | `(viewer_id, candidate_id, sport)` | One cached verdict per pair+sport |

**Secondary indexes** (query performance): FK columns used in hot filters are
indexed — `sessions.host_id/guest_id`, `game_invites.inviter_id/invitee_id/phase`,
`court_appointments.court_id/creator_id/scheduled_at`, `court_checkins.created_at`,
and the `status` columns on `user_reports` / `support_tickets`.

**Referential integrity:** every relationship is a real `FOREIGN KEY`. Owned
child collections (`sport_profiles`, `availability_slots`, `time_proposals`,
`appointment_participants`) use `cascade="all, delete-orphan"` so deleting the
parent cleans up its children.

## 5. Implementation & migrations

- **ORM:** SQLAlchemy models in `backend/models/` (one file per aggregate);
  `models/__init__.py` registers them all on `db.metadata`.
- **Schema creation:** `python seed.py` (dev) or `python manage.py init-db`
  (deploy) runs `db.create_all()` idempotently. `seed.py` loads 6 sample users,
  8 real Chicago courts, and 6 sessions.
- **Migrations:** Flask-Migrate (Alembic) is initialized with a baseline
  revision and two follow-ups in `backend/migrations/versions/` (add
  `users.is_admin`, add `users.bio_embedding`). See ADR-009 in `ARCHITECTURE.md`
  for the create-all-vs-migrations trade-off.
- **Dev → prod:** identical ORM; only `DATABASE_URL` changes
  (`sqlite:///rallypoint.db` → `postgresql://…`). `psycopg2-binary` is already a
  dependency.

The complete generated DDL is in **`docs/schema_ddl.sql`**.
